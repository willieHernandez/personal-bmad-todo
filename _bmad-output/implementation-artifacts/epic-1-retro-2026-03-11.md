## Successes & Strengths

1. **Architecture spec quality** — The detailed directory structure, naming conventions, query key patterns, and anti-patterns defined upfront eliminated ambiguity across all 7 stories and directly accelerated implementation velocity.
2. **Shared Zod schemas** — Single source of truth between client and server prevented API contract drift throughout the epic.
3. **Optimistic update pattern** — Established in Story 1.2 via TanStack Query and reused consistently in every subsequent mutation (create, update, delete, reorder, move). The app feels instant.
4. **Zero regressions** — Test count grew from 2 to 188 across 7 stories with no regressions. Each story added its own co-located tests.
5. **Previous Story Intelligence** — The Dev Notes and "Previous Story Intelligence" sections in each story file enabled cross-story learning, reducing version gotchas in later stories.
6. **100% FR delivery** — All 22 functional requirements assigned to Epic 1 were implemented and verified.

## Challenges & Growth Areas

1. **Code review fixes on every story (7/7)** — First-pass implementations consistently missed edge cases around keyboard interaction races, API response handling (204 responses, empty titles), and state management boundaries. Code reviews caught real defects every time.
2. **Keyboard binding conflict (Story 1.4 vs 1.6)** — Story 1.4 assigned Enter to "create sibling," Story 1.6 assigned Enter to "rename." This conflict was not caught during story creation and required mid-implementation rework. Sibling creation was moved to Ctrl+Enter.
3. **Framework version gotchas (4/7 stories)** — TanStack CLI ignoring `--add-ons` flag, `fastify-type-provider-zod` needing v6 for Zod v4, `react-resizable-panels` using `orientation` not `direction`, Tailwind v4 `group-hover` not generating CSS. Debug cycles spent on framework quirks.
4. **Story 1.4 required 3 rounds of fixes** — The tree view (most complex component) had empty title Zod rejections, missing 204 response handling in apiClient, and onBlur/onKeyDown race conditions requiring a cancelledRef pattern.
5. **Integration test gap** — Client and server are tested in isolation (mocked APIs on client, Fastify inject on server), but no tests exercise the full HTTP round-trip.

## Key Insights & Learnings

1. Architecture spec detail level directly correlates with implementation velocity — the more explicit the spec, the fewer decisions left to chance.
2. Code reviews are a critical quality gate, not ceremony — they caught real defects in every story and must remain mandatory.
3. Cross-story keyboard interaction specs must be validated before implementation to prevent mid-epic rework.
4. Test coverage is broad (188 tests) but shallow on integration — unit and route tests dominate, E2E and real integration tests are the gap.
5. The "Previous Story Intelligence" pattern in story files is highly effective — continue and expand it for Epic 2.
6. Technology version gotchas diminished over the epic as earlier learnings were documented — the system works when followed.
