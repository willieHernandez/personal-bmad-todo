## Exclusions

The following files are excluded from coverage metrics. Each exclusion has a rationale to ensure only meaningful, testable business logic is measured.

### Test files
- `**/*.test.{ts,tsx}`, `**/__tests__/**`
- **Reason:** Test files themselves are not application code.

### UI primitives (packages/client/src/components/ui/**)
- button.tsx, tabs.tsx, collapsible.tsx, separator.tsx, tooltip.tsx, resizable.tsx, scroll-area.tsx, breadcrumb.tsx, cn.ts
- **Reason:** Thin wrappers around shadcn/Radix UI library components. These are declarative pass-through components with no custom business logic. Testing them would validate third-party library behavior, not application code.

### Type-only files (packages/shared/src/types/**)
- node.types.ts, inbox.types.ts, session.types.ts, search.types.ts
- **Reason:** TypeScript type definitions produce no runtime code. They are compile-time constructs with zero executable statements.

### Generated files
- packages/client/src/routeTree.gen.ts
- **Reason:** Auto-generated by TanStack Router. Regenerated on every route change. Testing generated code is fragile and provides no value.

### Empty placeholder schemas
- packages/shared/src/schemas/inbox.schema.ts
- packages/shared/src/schemas/session.schema.ts
- packages/shared/src/schemas/search.schema.ts
- **Reason:** These files contain only placeholder comments (e.g., `// Inbox schemas — will be populated in a later story`). No executable code exists to cover.

### Entry points with side effects
- packages/client/src/main.tsx
- packages/server/src/db/index.ts
- **Reason:** These files execute side effects at import time (ReactDOM.render, SQLite database connection, filesystem directory creation). They cannot be unit-tested without mocking the entire runtime environment. They are validated through integration and E2E tests instead.

### Barrel / re-export files
- packages/shared/src/index.ts
- packages/client/src/lib/utils.ts
- **Reason:** These files only re-export symbols from other modules. They contain no logic. The actual implementations they re-export are covered by their own tests.

### Test utilities
- packages/server/src/db/test-db.ts
- **Reason:** Test infrastructure code, not application code. Used only within test suites to set up in-memory databases.

### Integration test files
- packages/server/src/integration/**
- **Reason:** Integration tests run under a separate configuration (`vitest.integration.config.ts`) and are not part of the unit test suite.

## Configuration

Coverage is configured in `vitest.config.ts` at the root workspace level using the v8 provider. Thresholds are enforced — the test suite will fail if any metric drops below 70%.

```
pnpm test:unit:coverage    # Run tests with coverage
```

HTML report is generated at `./coverage/` for detailed per-file and per-line inspection.
